# Lumenci Assistant — User Flow Steps (locked so far)

Working list of user flow steps, being built incrementally before drafting the flow diagram. Steps below are locked; more will be appended as we continue.

1. **Session starts.** Opening the app creates a new session automatically; every subsequent action (upload, generate, chat, export) is scoped to this session.
2. **Upload claim chart.** Analyst clicks "Upload Claim Chart" and selects the CSV — a 3-column file (Patent Claim Element / Accused Product Feature (Evidence) / AI Reasoning), pre-populated by the analyst's prior process.
3. **Upload evidence/product docs.** Analyst clicks "Upload Evidence" and selects one or more `.txt` files and/or supplies a link. This action is repeatable — the analyst can add more sources in additional batches before generating.
4. **Setup screen (system prompt).** An editable freeform text area where the analyst can add their own instructions (e.g., how conservative/aggressive to be, what counts as strong evidence). Editable before Generate on the setup screen, and remains editable after Generate via a settings button in the chat panel — every LLM call reads the session's current freeform text live, so an edit made mid-session only affects proposals/classifications from that point forward, never rewriting past ones. This freeform text is layered on top of a **hidden backend baseline system prompt** — not shown to or editable by the analyst — which defines core mechanics (confidence-tier rubric, chart-rendering/output-format contract, etc.). The baseline is worded to take precedence over the analyst's freeform text if the two ever conflict, so the analyst can extend AI behavior but cannot neutralize the core mechanics the UI depends on.
5. **Generate.** Analyst clicks "Generate" — enabled once the chart is uploaded; evidence is optional at this point (rows with none simply classify Weak). This triggers the AI reasoning pass over the uploaded chart, tagging every row with a confidence tier (Strong/Moderate/Weak).
6. **Split view loads.** Once Generate completes, the claim chart renders (with confidence tags) in the left panel; the chat window appears in the right panel.
7. **Opening chat message.** The AI also posts an unprompted opening message in the chat, flagging which rows are Weak/Moderate — it flags only, it does not pre-draft fixes for them unprompted.
8. **Analyst sends a message.** The analyst can type a message with no row selected. Clicking a row first (inserting the `@Row` chip) is a convenience for speed/precision, not a requirement.
9. **Referent resolution.** The AI has full chart content as context and uses its own understanding to figure out which row a message refers to. If it isn't confident which row is meant, it simply asks the analyst a clarifying question to confirm — no forced click, no separate matching mechanism.
10. **AI proposes a change.** Once the AI is confident which row it's addressing, it proposes a change — its reasoning/proposal is shaped by what the analyst actually asked for in their message about that row, not a generic re-evaluation.
11. **Pending state.** The proposal is shown as a pending edit directly in that row's chart cell, with **Accept**, **Reject**, and **Modify** controls next to the row.
12. **Accept.** The chart cell updates to the new value; the row's previous value is saved as the single undo slot; the pending state clears.
13. **Reject.** The chart cell reverts to showing the current (unchanged) value; the pending state clears; nothing is saved to the undo slot since nothing was applied.
14. **Modify.** Clicking Modify inserts that row's `@Row` chip into the chat input; the analyst then types their feedback/instructions as a normal chat message (same mechanic as directly clicking a row). The AI's revised proposal goes back into the same pending state on the chart — it requires another explicit Accept (or Reject/Modify again) before it touches the chart. Every proposal, first or revised, always requires explicit Accept; there is no "auto-apply after Modify" behavior.
15. **Undo.** A persistent button next to every row, enabled whenever a previous applied (accepted) value exists for that row, disabled/no-op otherwise. Clicking Undo shows a confirmation prompt; confirming reverts the row to its previous value and clears the undo slot (single-step undo — a second consecutive undo has nothing left to revert to).
16. **No evidence found (edge case).** Instead of proposing a change, the AI states it found no supporting evidence in the uploaded docs for that row and asks the analyst, in chat, for more evidence — either re-uploading an evidence document or providing a URL.
17. **New evidence supplied.** The analyst re-uploads a document or provides a URL (same "Upload Evidence" action/endpoint as step 3, usable again mid-session). A re-uploaded document is ingested as text; a URL is fetched by the backend (HTTP GET + HTML-to-text extraction, single page, no crawling/JS rendering). Either way, the result is treated as a new evidence source, added to the session's evidence pool.
18. **AI retries the proposal.** With the new evidence source available, the AI attempts the proposal again — re-entering the normal pending flow (step 11).
19. **Flag evidence as incorrect (edge case).** A dedicated flag icon sits next to every row, separate from Accept/Reject/Modify/Undo. Clicking it is what deterministically triggers the backend to re-scan the session's evidence pool for that claim element — this trigger does not rely on the AI detecting "wrong"/"incorrect" language in a chat message, since that detection could be missed and silently skip the safeguard.
20. **Re-grounded correction.** After the flag click, that row's `@Row` chip appears in chat so the analyst can describe what's wrong. The AI's corrected response must quote the exact source line found during the re-scan. The correction re-enters the normal pending flow (step 11) — it requires explicit Accept like any other proposal.
21. **Export.** An "Export to Word" action is always available once Generate has run, with no other precondition — the analyst does not need to resolve pending proposals first. Export generates a `.docx` from the chart's exact current state at that moment; any pending (not-yet-accepted) cell exports as its current, unaccepted value, same as if no proposal had been made on it.
